## Context

`apps/web/lib/corpus-client.ts` is the playground's client-side bridge to the engine. Because Next.js client components cannot use `node:fs`, every verb the browser engine sees must be statically imported. The current file lists 20 verbs:

```ts
import bej from '../../../data/verbs/bej.json';
import djeg from '../../../data/verbs/djeg.json';
// ... 18 more
const CORPUS: VerbEntry[] = [bej, djeg, /* ... */];
```

This was correct when the corpus had ~20 verbs. The corpus is now 204 verbs (verified by `data/verbs/index.json` and `verify-engine.ts`). Adding 184 more `import` lines is feasible but ugly and easy to forget when ingesting new verbs.

A single bundled JSON eliminates the maintenance hazard: the build step that already enumerates every valid verb file emits a derived file the client can import once.

## Goals / Non-Goals

**Goals:**

- Every corpus verb is reachable from the playground.
- Adding a new verb to `data/verbs/<lemma>.json` requires no manual edit to client code; `npm run build:corpus` propagates it.
- Client API (`ensureClientConfigured`, `findClientEntry`) unchanged.
- Bundle-size regression is bounded and predictable.

**Non-Goals:**

- No lazy loading.
- No mutation of `index.json` shape.
- No client-side fetch from `/api/verbs/_full` (which doesn't exist and would add an HTTP request we don't need).

## Decisions

### D1. Single bundled JSON, generated by `build:corpus`

`scripts/build-corpus.ts` already loads, validates, and round-trips every verb. Adding one more `writeFileSync` to emit a serialized array is the cheapest change.

The output file is `data/verbs/_corpus.client.json`. Naming convention: leading underscore distinguishes it from per-verb files; suffix `.client.json` clarifies its consumer. Sibling derived files (`index.json`, `version.json`) follow the same directory.

The file content is the raw `VerbEntry[]` array (ordered by `id`). Pretty-printed (matching `index.json`) for diff readability. Approximate size: 204 entries × ~700 bytes ≈ 140 KB raw, ~50 KB gzipped.

### D2. corpus-client.ts simplification

```ts
import { configure, type VerbEntry } from '@foljapp/engine';

import corpusData from '../../../data/verbs/_corpus.client.json';
import versionData from '../../../data/verbs/version.json';

const CORPUS = corpusData as VerbEntry[];

let configured = false;

export function ensureClientConfigured(): void {
  if (configured) return;
  configure(CORPUS, (versionData as { version: string }).version);
  configured = true;
}

ensureClientConfigured();

export function findClientEntry(lemma: string): VerbEntry | undefined {
  return CORPUS.find((e) => e.lemma === lemma);
}
```

The 20 individual imports go away. The `CORPUS` array, `ensureClientConfigured()`, and `findClientEntry()` keep their semantics.

Type-cast (`as VerbEntry[]`) bridges the JSON-shape import (which TS sees as a structural literal) to the engine's `VerbEntry` type. The build-corpus gate already validates each entry against `verbEntrySchema`, so the cast is sound at runtime.

### D3. Version constant in build-corpus.ts

Currently `emitVersion` hardcodes `version: '0.1.0'`. The hand-bumped `data/verbs/version.json` is at `0.1.1`. Running `build:corpus` would silently revert the bump.

Lift the version into a `const CORPUS_VERSION = '0.1.1'` declared near the top of `build-corpus.ts`. Future corpus bumps update this one line.

### D4. .gitignore policy

Both `index.json` and `version.json` are currently checked in. Treat `_corpus.client.json` the same — check it in. Reasoning:

- Already checked-in derived files set the precedent.
- Reviewing PRs, a corpus diff in the bundled file makes verb-add/remove visible to reviewers.
- Avoids requiring contributors to run `build:corpus` before `npm run dev` works.

### D5. Accidental drift between bundled file and per-verb files

If someone hand-edits a verb JSON without rerunning `build:corpus`, the bundled file diverges. Mitigations:

- The CI's `npm run build` runs `build:corpus` (verify? — at minimum there's a round-trip gate). Add a CI step that runs `build:corpus` and fails if the bundled file is dirty (`git diff --exit-code data/verbs/_corpus.client.json`).
- Document in `CLAUDE.md` (or this design.md) that hand-editing a verb file requires `build:corpus` to refresh derived files.

For now: rely on developer discipline + CI check. If drift becomes a recurring issue, swap to runtime generation (server reads per-verb files at build, emits bundle then).

### D6. Bundle-size tradeoff

The playground's First Load JS goes from 150 KB to ~200–230 KB (per the build report). At 200 KB this is squarely in the "acceptable single-page-app initial payload" range; the playground is a known heavy reference page. Lazy loading would save ~50 KB on first paint but add latency on verb-pick interactions, which is the wrong tradeoff for a tool whose main interaction IS picking verbs.

## Tradeoffs

- **Bundle size vs. interactivity**: bundle wins. ~50 KB gzipped is cheap; sub-second verb switches matter more.
- **Generated file vs. import sprawl**: bundled file wins. Less code, less drift.
- **Checked-in derived file vs. .gitignore**: checked-in wins for review visibility, matching the existing pattern.
